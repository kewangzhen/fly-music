# Fly-Music 数据库设计问题与解决方案

## 一、数据库基础问题

### 问题1：如何设计用户表支持多种角色
**问题描述：** 系统需要支持普通用户、VIP用户、管理员等多种角色，如何在用户表中体现？
**解决方案：** 添加 `role` 字段区分角色，0=普通用户，1=VIP用户，2=管理员。VIP用户的过期时间用 `vip_expire_at` 字段记录。

```sql
ALTER TABLE users ADD COLUMN role TINYINT DEFAULT 0 COMMENT '角色: 0普通用户,1VIP,2管理员';
ALTER TABLE users ADD COLUMN vip_expire_at TIMESTAMP NULL COMMENT 'VIP过期时间';
```

### 问题2：歌曲与歌手的多对多关系如何设计
**问题描述：** 一首歌可能有多个歌手，一个歌手可能有多首歌，如何设计关联关系？
**解决方案：** 创建歌曲歌手关联表 `song_artists`，存储歌曲ID和歌手ID的对应关系。

```sql
CREATE TABLE song_artists (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    song_id BIGINT NOT NULL,
    artist_id BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (song_id) REFERENCES songs(id),
    FOREIGN KEY (artist_id) REFERENCES artists(id),
    UNIQUE KEY uk_song_artist (song_id, artist_id)
);
```

### 问题3：歌单与歌曲的多对多关系如何设计
**问题描述：** 用户创建的歌单可以包含多首歌，一首歌也可以被多个歌单收藏。
**解决方案：** 创建歌单歌曲关联表 `playlist_songs`。

```sql
CREATE TABLE playlist_songs (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    playlist_id BIGINT NOT NULL,
    song_id BIGINT NOT NULL,
    position INT DEFAULT 0 COMMENT '在歌单中的位置',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (playlist_id) REFERENCES playlists(id),
    FOREIGN KEY (song_id) REFERENCES songs(id),
    UNIQUE KEY uk_playlist_song (playlist_id, song_id)
);
```

## 二、查询性能问题

### 问题4：用户播放历史数据量太大查询慢
**问题描述：** 随着用户播放次数增加，播放历史表数据量快速增长，查询变慢。
**解决方案：** 
- 对 `user_id` 和 `played_at` 字段建立联合索引
- 定期归档历史数据，将超过1年的数据迁移到归档表
- 查询最近播放时使用分页，只查询必要字段

```sql
CREATE INDEX idx_user_played ON play_history(user_id, played_at DESC);

-- 归档存储过程示例
CREATE PROCEDURE archive_old_history()
BEGIN
    INSERT INTO play_history_archive
    SELECT * FROM play_history
    WHERE played_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);
    
    DELETE FROM play_history
    WHERE played_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);
END;
```

### 问题5：热门歌曲排行榜查询效率低
**问题描述：** 按播放次数排序查询热门歌曲，需要对全表扫描。
**解决方案：** 
- 使用物化视图或定时任务生成排行榜缓存表
- 预先计算热门歌曲的排名，定时刷新
- 使用 Redis 缓存热门数据

```sql
-- 创建热门歌曲缓存表
CREATE TABLE hot_rankings (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    song_id BIGINT NOT NULL,
    play_count BIGINT DEFAULT 0,
    rank_position INT DEFAULT 0,
    calculated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (song_id) REFERENCES songs(id)
);

-- 定时更新热门排行（每天凌晨执行）
CREATE EVENT daily_hot_ranking
ON SCHEDULE EVERY 1 DAY
DO
BEGIN
    INSERT INTO hot_rankings (song_id, play_count, rank_position)
    SELECT song_id, COUNT(*) as play_count,
           @row_num:=@row_num+1 as rank
    FROM play_history
    WHERE played_at > DATE_SUB(NOW(), INTERVAL 7 DAY)
    GROUP BY song_id
    ORDER BY play_count DESC
    LIMIT 100;
END;
```

### 问题6：用户收藏查询需要关联多张表
**问题描述：** 查询用户收藏的歌曲需要关联用户表、收藏表、歌曲表、歌手表，性能较差。
**解决方案：** 
- 为常用查询创建索引
- 使用预加载（Eager Loading）减少查询次数
- 考虑冗余存储，收藏表中直接存储歌曲名称、歌手名称等常用字段

```sql
-- 创建复合索引
CREATE INDEX idx_favorite_user_target ON user_favorites(user_id, target_type, target_id);

-- 冗余字段设计示例
ALTER TABLE user_favorites ADD COLUMN target_title VARCHAR(255);
ALTER TABLE user_favorites ADD COLUMN target_cover VARCHAR(255);
```

## 三、数据一致性

### 问题7：删除歌曲时如何处理关联数据
**问题描述：** 删除歌曲时，播放历史、收藏、歌单中的歌曲记录如何处理？
**解决方案：** 
- 逻辑删除：添加 `deleted_at` 字段，软删除歌曲
- 级联处理：删除歌曲时同时删除关联的历史记录
- 保留数据：使用外键的 ON DELETE SET NULL

```sql
-- 逻辑删除方案
ALTER TABLE songs ADD COLUMN deleted_at TIMESTAMP NULL;
ALTER TABLE songs ADD INDEX idx_deleted (deleted_at);

-- 外键级联删除示例
ALTER TABLE play_history
ADD CONSTRAINT fk_history_song
FOREIGN KEY (song_id) REFERENCES songs(id) ON DELETE CASCADE;
```

### 问题8：用户关注关系的数据一致性
**问题描述：** 关注和被关注的数量需要保持一致，如何保证？
**解决方案：** 
- 不存储冗余的计数字段，每次通过查询计算
- 使用数据库事务保证一致性
- 定期校验数据一致性

```sql
-- 查询粉丝数量（不存储冗余）
SELECT COUNT(*) FROM user_follows WHERE followed_id = ?;

-- 使用事务保证一致性
START TRANSACTION;
INSERT INTO user_follows (follower_id, followed_id) VALUES (?, ?);
-- 不在这里更新计数，依赖查询计算
COMMIT;
```

## 四、推荐系统相关

### 问题9：如何存储用户的兴趣标签
**问题描述：** 系统需要根据用户的听歌历史生成兴趣标签，如何存储？
**解决方案：** 创建用户兴趣画像表，存储用户的音乐偏好标签和权重。

```sql
CREATE TABLE user_interest_profiles (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    category_id BIGINT,
    genre VARCHAR(50),
    mood VARCHAR(50),
    artist_id BIGINT,
    weight DOUBLE DEFAULT 0.0 COMMENT '兴趣权重 0-1',
    calculated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (category_id) REFERENCES categories(id),
    FOREIGN KEY (artist_id) REFERENCES artists(id),
    INDEX idx_user_interest (user_id, weight DESC)
);
```

### 问题10：如何记录用户行为用于推荐
**问题描述：** 推荐系统需要分析用户行为，如何设计行为记录表？
**解决方案：** 创建用户行为记录表，记录用户的播放、收藏、搜索等行为。

```sql
CREATE TABLE user_behaviors (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    behavior_type VARCHAR(20) NOT NULL COMMENT 'play/favorite/search/skip/share',
    target_type VARCHAR(20) NOT NULL COMMENT 'song/playlist/album/artist',
    target_id BIGINT NOT NULL,
    duration INT COMMENT '停留时长(秒)',
    completed BOOLEAN DEFAULT TRUE COMMENT '是否完整播放',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id),
    INDEX idx_user_behavior (user_id, behavior_type, created_at),
    INDEX idx_behavior_target (target_type, target_id)
);
```

## 五、特殊业务场景

### 问题11：AI生成音乐如何存储
**问题描述：** AI音乐实验室功能需要存储用户生成的AI音乐，如何设计？
**解决方案：** 创建AI音乐生成记录表，存储生成参数和结果。

```sql
CREATE TABLE ai_music_generations (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    prompt TEXT COMMENT '用户输入的生成提示',
    genre VARCHAR(50) COMMENT '音乐风格',
    mood VARCHAR(50) COMMENT '情绪',
    music_url VARCHAR(500) COMMENT '生成的音乐URL',
    cover_url VARCHAR(500) COMMENT '封面图URL',
    status VARCHAR(20) DEFAULT 'processing' COMMENT 'processing/completed/failed',
    duration INT COMMENT '时长(秒)',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    completed_at TIMESTAMP NULL,
    FOREIGN KEY (user_id) REFERENCES users(id),
    INDEX idx_user_status (user_id, status)
);
```

### 问题12：歌曲热度分数如何计算和存储
**问题描述：** 排行榜需要根据多维度计算歌曲热度，如何设计？
**解决方案：** 创建歌曲热度分数表，存储多维度的热度评分。

```sql
CREATE TABLE song_hot_scores (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    song_id BIGINT NOT NULL,
    play_score DOUBLE DEFAULT 0 COMMENT '播放得分',
    favorite_score DOUBLE DEFAULT 0 COMMENT '收藏得分',
    share_score DOUBLE DEFAULT 0 COMMENT '分享得分',
    recent_score DOUBLE DEFAULT 0 COMMENT '近期热度得分',

    FOREIGN KEY (song_id) REFERENCES songs(id),
    INDEX idx_total_score (total_score DESC),
    INDEX idx_calculated (calculated_at)
);
```

### 问题14：防止SQL注入
**问题描述：** 如何防止SQL注入攻击？
**解决方案：** 
- 使用参数化查询（PreparedStatement）
- 避免字符串拼接SQL
- 限制数据库用户权限

```java
// 错误写法（SQL注入风险）
String sql = "SELECT * FROM users WHERE username = '" + username + "'";

// 正确写法（参数化查询）
@Query("SELECT u FROM User u WHERE u.username = :username")
User findByUsername(@Param("username") String username);
```

### 问题16：分类层级如何设计
**问题描述：** 音乐分类有多级层级（如：流行→华语流行），如何设计？
**解决方案：** 使用邻接表模型（自关联）或路径枚举。

```sql
-- 邻接表模型
CREATE TABLE categories (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    parent_id BIGINT NULL DEFAULT NULL,
    level INT DEFAULT 1 COMMENT '层级深度',
    FOREIGN KEY (parent_id) REFERENCES categories(id)
);

-- 路径枚举（冗余存储层级路径）
ALTER TABLE categories ADD COLUMN path VARCHAR(500) COMMENT '路径: 0/1/5';
-- 示例: 0/1/5 表示 音乐/流行/华语流行
```
