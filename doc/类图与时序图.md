## 4. 推荐模块

### 4.1 类图

**类关系说明：**

本模块采用标准的 MVC + Repository 分层架构设计，类与类之间的关系采用面向接口编程和依赖注入的设计模式。

**继承关系**：RecommendationServiceImpl 实现了 RecommendationService 接口。接口定义了业务方法规范，实现类负责具体的业务逻辑实现。这种设计遵循了面向接口编程的原则，使得业务逻辑的实现可以被替换，便于后期扩展和单元测试。

**依赖关系**：RecommendationController 依赖 RecommendationService 接口，而不直接依赖实现类。控制层通过接口调用服务层，实现了控制层与服务层的解耦。当服务实现类发生变更时，控制器层无需修改，降低了代码耦合度。

**关联关系**：RecommendationServiceImpl 通过依赖注入的方式关联了多个组件，包括 UserBehaviorRepository、UserInterestProfileRepository、RadarPlaylistRepository、SongRepository、UserRepository 以及 ObjectMapper。这些组件分别负责用户行为数据持久化、用户兴趣画像持久化、雷达歌单持久化、歌曲数据查询、用户数据查询以及 JSON 序列化反序列化功能。

**实体继承**：UserBehavior、RadarPlaylist、UserInterestProfile 等实体类都继承自 JPA 的 BaseEntity（隐式），包含公共字段 id、createdAt、updatedAt。

**数据传输**：服务实现类依赖 DTO（Data Transfer Object）进行数据传输，包括 RadarPlaylistDTO、SongDTO、UserInterestDTO。DTO 用于封装响应数据，与前端进行数据交换，同时隐藏内部实体结构。

**数据流向**：请求流程从客户端发起，经过 Controller 接收请求，调用 Service 层处理业务逻辑，Service 层调用 Repository 进行数据持久化操作，最终到达数据库。响应流程则相反，从数据库查询数据，经过 Entity 封装，Repository 返回，Service 处理后转换为 DTO，最后由 Controller 返回给客户端。

**各层职责**：控制层负责接收并解析客户端 HTTP 请求、参数校验、调用 Service 层方法、封装响应结果以及处理异常。服务层接口定义业务接口规范和声明业务方法签名，不涉及具体实现。服务层实现类负责实现核心业务逻辑、处理业务规则、协调多个 Repository、事务管理以及异常处理。持久化层负责数据增删改查、定义数据访问方法和 JPA 查询方法。实体层负责映射数据库表、定义实体属性和表关系。数据传输层负责封装响应数据、数据格式转换和隐藏内部结构。

### 4.4 时序图 - 生成心动歌单（详细流程）

**流程说明：**

该时序图展示了根据用户历史行为生成个性化心动歌单的完整流程，整个流程分为四个主要步骤。

步骤一为获取用户行为。系统首先通过 UserBehaviorRepository 从数据库中查询指定用户的所有行为记录，这些行为记录包括用户对歌曲的播放、收藏、分享、评论等操作。如果查询结果为空，说明该用户没有任何行为数据，系统将使用默认的兴趣标签（流行、摇滚、民谣）作为推荐依据。如果存在行为数据，系统将获取到用户最近一段时间内的所有行为记录，这些记录将用于后续的兴趣分析。

步骤二为计算兴趣权重。系统首先对用户行为数据按歌曲进行分组，计算每首歌曲的总得分。每首歌曲的得分由用户对该歌曲的各种行为权重累加得到，播放行为权重为1，完整收听权重为2，收藏权重为5，分享权重为3。然后，系统遍历每首歌曲，通过 SongRepository 查询歌曲的详细信息，包括歌曲所属的分类。根据歌曲的分类信息，将歌曲得分累加到对应的分类上，形成用户对各个分类的兴趣得分。完成所有歌曲的分类得分统计后，系统对分类得分进行排序，选取得分最高的前三个分类作为用户的核心兴趣标签。例如最终可能得到摇滚占45%、流行占30%、民谣占15%的兴趣分布。确定兴趣分布后，系统将更新用户兴趣画像，保存到 UserInterestProfileRepository 中。

步骤三为根据兴趣选歌。系统根据步骤二确定的 Top3 兴趣分类，依次从每个分类中选取热门歌曲。对于每个兴趣分类，系统首先通过 CategoryRepository 查询分类信息，然后通过 SongRepository 查询该分类下按播放量排序的歌曲列表，选取前50首热门歌曲。三个分类分别选取后，系统将三个列表合并，并进行去重处理，最终从约150首歌曲中精选出20首推荐歌曲，形成心动歌单的歌曲列表。

步骤四为保存歌单。系统根据前面步骤确定的歌曲列表和兴趣画像，构建 RadarPlaylist 实体对象。实体对象包含用户ID、歌曲ID列表（JSON格式）、兴趣摘要信息（包含最高兴趣标签及其得分分布）、创建时间和过期时间。过期时间设置为当前时间的后一天，即次日过期，确保歌单每天都会更新。最后将构建好的雷达歌单实体保存到 RadarPlaylistRepository 中，完成整个心动歌单生成流程。

## 5. 雷达歌单模块

### 5.1 类图

**类关系说明：**

本雷达歌单模块是推荐系统的核心组成部分，专门用于生成和管理用户的个性化心动歌单。该模块采用 MVC + Repository 分层架构设计，同时引入了定时任务调度机制，类与类之间的关系采用面向接口编程和依赖注入的设计模式。

继承关系：RadarPlaylistServiceImpl 实现了 RadarPlaylistService 接口。接口定义了雷达歌单业务方法规范，实现类负责具体的歌单生成、查询和定时更新逻辑。这种设计遵循了面向接口编程的原则，使得业务逻辑的实现可以被替换，便于后期扩展和单元测试。

依赖关系：RadarPlaylistController 依赖 RadarPlaylistService 接口，而不直接依赖实现类。控制层通过接口调用服务层，实现了控制层与服务层的解耦。当服务实现类发生变更时，控制器层无需修改，降低了代码耦合度。

关联关系：RadarPlaylistServiceImpl 通过依赖注入的方式关联了多个组件，包括 RadarPlaylistRepository（雷达歌单持久化）、UserBehaviorRepository（用户行为数据查询）、UserInterestProfileRepository（用户兴趣画像持久化）、SongRepository（歌曲数据查询）、CategoryRepository（分类数据查询）、UserRepository（用户数据查询）以及 ObjectMapper（JSON序列化）。这些组件分别负责雷达歌单的持久化、用户行为分析、兴趣画像管理、歌曲查询、分类查询、用户数据查询以及歌单数据的序列化和反序列化功能。服务实现类还引入了定时任务调度器（Scheduler），用于每日自动为所有活跃用户生成雷达歌单。

实体继承：RadarPlaylist 实体类继承自 JPA 的 BaseEntity（隐式），包含公共字段 id、createdAt、updatedAt。实体类映射数据库中的 radar_playlists 表，存储用户的雷达歌单信息，包括用户ID、歌单名称、描述、封面URL、歌曲ID列表（JSON格式）、兴趣摘要信息（JSON格式）、歌曲数量、类型、过期时间等字段。UserInterestProfile 实体类同样继承自 BaseEntity，存储用户的兴趣画像数据，包括用户ID、曲风偏好（JSON格式）、艺术家偏好（JSON格式）、情绪偏好（JSON格式）和更新时间。

数据传输：控制层接收客户端请求参数，包括 userId（用户ID）等。服务层生成 RadarPlaylist 实体和 RadarPlaylistDTO 用于返回给客户端。RadarPlaylistDTO 是专门用于前端展示的数据传输对象，包含歌单ID、用户ID、歌曲列表（SongDTO列表）、兴趣摘要、创建时间和过期时间。SongDTO 用于封装单个歌曲的简要信息，包括歌曲ID、标题、艺术家、封面和时长。

数据流向：请求流程从客户端发起，GET 请求到 /api/radar/{userId} 或 /api/radar 接口，RadarPlaylistController 接收请求后，调用 RadarPlaylistService 的相应方法处理业务逻辑。服务层首先从 RadarPlaylistRepository 查询用户是否已有未过期的雷达歌单，如果有则直接返回缓存数据，如果没有则调用歌单生成方法创建新的雷达歌单。歌单生成过程中会分析用户的历史行为数据，计算兴趣标签，从各分类选取推荐歌曲，最后保存到数据库。响应流程则相反，从数据库查询数据，经过 Entity 封装，Repository 返回，Service 处理后转换为 DTO，最后由 Controller 返回给客户端。

各层职责：控制层负责接收并解析客户端 HTTP 请求、参数校验、调用 Service 层方法、封装响应结果以及处理异常。服务层接口定义业务接口规范和声明业务方法签名，不涉及具体实现。服务层实现类负责实现核心业务逻辑，包括雷达歌单查询、歌单生成、兴趣分析、歌曲推荐。服务实现类还包含定时任务方法，用于每日凌晨自动为所有活跃用户生成雷达歌单，并清理过期的歌单数据。持久化层负责 RadarPlaylist、UserInterestProfile 实体的增删改查操作。实体层负责映射数据库表、定义实体属性。

### 5.2 时序图 - 获取用户雷达歌单

**流程说明：**

该时序图展示了客户端获取用户个性化雷达歌单的完整流程，整个流程分为缓存检查、歌单生成（如果需要）和响应返回三个主要阶段。

缓存检查阶段：客户端发起 GET 请求到 /api/radar/{userId} 接口，请求获取指定用户的雷达歌单。RadarPlaylistController 接收请求后，提取用户ID参数，调用 RadarPlaylistServiceImpl 的 getUserRadarPlaylist 方法处理业务逻辑。服务层首先通过 RadarPlaylistRepository 的 findByUserIdAndExpiresAtAfter 方法查询该用户是否存在未过期的雷达歌单。查询时使用当前时间作为过期时间参数，系统会返回过期时间大于当前时间的歌单记录。

歌单生成阶段：如果查询结果为空，说明该用户没有有效的雷达歌单（可能从未生成过，或者歌单已过期），系统需要调用 generateRadarPlaylist 方法为用户生成新的雷达歌单。生成过程首先通过 UserBehaviorRepository 查询用户最近一段时间内的所有行为记录，包括播放、收藏、分享、评论等操作。如果用户没有任何行为数据，系统将使用默认的兴趣标签（流行、摇滚、民谣）作为推荐依据。如果存在行为数据，系统会对行为数据按歌曲进行分组，计算每首歌曲的总得分，然后遍历每首歌曲查询其所属分类，累加各分类的得分，形成用户对各个分类的兴趣得分。排序后选取得分最高的前三个分类作为用户的核心兴趣标签。根据确定的兴趣分类，分别从各分类中选取热门歌曲，每个分类选取前50首，然后合并去重，最终得到20首推荐歌曲。构建 RadarPlaylist 实体对象，包含用户ID、歌单名称（"今日推荐"）、歌曲ID列表（JSON格式）、兴趣摘要信息（JSON格式，包含Top标签和各分类得分）、歌曲数量（20首）、类型（0-每日推荐）、过期时间（次日）。最后将构建好的雷达歌单保存到 RadarPlaylistRepository 中。

## 6. AI曲风识别模块

### 6.1 类图

**类关系说明：**

本AI曲风识别模块是音乐管理系统的重要组成部分，用于通过AI技术分析用户上传的音频文件，自动识别音乐的曲风类型。该模块采用 MVC + Repository 分层架构设计，同时引入了异步任务处理和外部AI服务调用机制，类与类之间的关系采用面向接口编程和依赖注入的设计模式。

继承关系：GenreRecognitionServiceImpl 实现了 GenreRecognitionService 接口。接口定义了曲风识别业务方法规范，实现类负责具体的音频分析、曲风分类、歌曲分类更新等逻辑。这种设计遵循了面向接口编程的原则，使得业务逻辑的实现可以被替换，便于后期扩展和单元测试。

依赖关系：GenreRecognitionController 依赖 GenreRecognitionService 接口，而不直接依赖实现类。控制层通过接口调用服务层，实现了控制层与服务层的解耦。当服务实现类发生变更时，控制器层无需修改，降低了代码耦合度。

关联关系：GenreRecognitionServiceImpl 通过依赖注入的方式关联了多个组件，包括 SongRepository（歌曲数据持久化和更新）、CategoryRepository（分类数据查询）、FileStorageService（音频文件存储）、RestTemplate（HTTP客户端，用于调用外部AI曲风识别服务）以及配置属性（apiKey、apiProvider、genreRecognitionEnabled、genreRecognitionServiceUrl等）。这些组件分别负责歌曲数据的持久化和分类更新、分类数据的查询、音频文件的上传和存储、调用外部AI服务进行曲风分析等功能。服务实现类定义了支持的曲风标签列表，包括流行、摇滚、民谣、古典、爵士、电子、说唱、R&B、轻音乐、乡村、蓝调、嘻哈、金属、舞曲、抒情等常见音乐风格。

实体继承：Song 实体类继承自 JPA 的 BaseEntity（隐式），包含公共字段 id、createdAt、updatedAt。Song 实体类映射数据库中的 songs 表，其中 categoryId 字段用于存储歌曲所属的分类（曲风）。Category 实体类映射 categories 表，存储曲风分类信息，包括分类名称、描述、层级等。

数据传输：控制层接收客户端请求参数，包括 songId（歌曲ID）或 audioFile（音频文件）等。服务层通过调用外部AI服务获取曲风分析结果，然后将识别结果更新到歌曲记录中，并返回 GenreRecognitionResultDTO 用于给前端展示。GenreRecognitionResultDTO 是专门用于展示曲风识别结果的数据传输对象，包含歌曲ID、识别的主要曲风、曲风置信度得分、各曲风候选列表等信息。

数据流向：请求流程从客户端发起，POST 请求到 /api/genre/recognize 接口（上传音频文件）或 GET 请求到 /api/genre/recognize/{songId} 接口（识别已有歌曲），GenreRecognitionController 接收请求后，调用 GenreRecognitionService 的相应方法处理业务逻辑。服务层首先获取需要识别的音频文件，然后调用外部AI曲风识别服务进行分析。外部服务返回曲风分析结果后，服务层将识别结果更新到歌曲记录的 categoryId 字段中。如果识别结果置信度较高，系统还会建议用户将歌曲归入对应的分类。

各层职责：控制层负责接收并解析客户端 HTTP 请求、参数校验、调用 Service 层方法、封装响应结果以及处理异常。服务层接口定义业务接口规范和声明业务方法签名，不涉及具体实现。服务层实现类负责实现核心业务逻辑，包括获取音频文件、调用外部AI服务进行曲风分析、解析识别结果、更新歌曲分类信息。持久化层负责 Song、Category 实体的增删改查操作。实体层负责映射数据库表、定义实体属性。

### 6.2 时序图 - 上传音频识别曲风

**流程说明：**

该时序图展示了用户上传音频文件并通过AI识别曲风的完整流程，整个流程分为上传音频、调用AI服务、分析结果和返回结果四个主要阶段。

获取行为数据阶段：客户端发起 GET 请求到 /api/genre/recognize/{userId} 接口，请求识别指定用户的曲风偏好。GenreRecognitionController 接收请求后，提取用户ID参数，调用 GenreRecognitionServiceImpl 的 recognizeUserGenre 方法处理业务逻辑。服务层首先通过 UserBehaviorRepository 的 findByUserId 方法查询该用户所有的行为记录，包括播放、收藏、分享、评论等操作。如果用户没有任何行为数据，系统将使用默认曲风列表（流行、摇滚、民谣、古典、爵士、电子、说唱、R&B、轻音乐、乡村）作为基础，每个曲风给予均等的初始得分。

计算曲风得分阶段：如果用户存在行为数据，系统需要分析这些行为来计算曲风偏好得分。服务层首先遍历每条行为记录，通过 SongRepository 根据歌曲ID查询歌曲的详细信息，包括歌曲所属的分类（曲风）。然后根据用户对该歌曲的行为类型（播放、收藏、分享、评论）获取对应的权重分数。系统定义了行为权重映射：播放权重为1，完整收听权重为2，收藏权重为5，分享权重为3，评论权重为2。将每条行为的权重分数累加到对应的曲风分类上，形成用户对各个曲风的原始得分。完成所有行为记录的分析后，系统对各曲风得分进行归一化处理，将得分转换为0到100之间的数值，便于前端展示和比较。排序后选取得分最高的前5个曲风作为用户的主要偏好曲风。

更新兴趣画像阶段：系统根据计算得出的曲风得分，更新用户的兴趣画像。首先通过 UserInterestProfileRepository 查询用户是否已存在兴趣画像记录。如果已存在，则更新现有的画像记录，替换曲风偏好数据。如果不存在，则创建新的画像记录。画像数据以 JSON 格式存储在 genrePreferences 字段中，包括各曲风名称和对应的得分。保存完成后，实体对象的 updatedAt 字段会自动更新为当前时间。

返回结果阶段：服务层将更新后的兴趣画像转换为 UserInterestDTO 格式返回给客户端。DTO 包含用户ID、曲风偏好得分Map（Key为曲风名称，Value为0-100的得分）、推荐的曲风列表（Top5）、艺术家偏好列表、情绪偏好Map和画像更新时间。Controller 最终返回 200 OK 响应给客户端，响应体中包含 code 为 200 和 data 为兴趣画像数据的 JSON 对象。

### 7.1 类图

**类关系说明：**

本音乐播放与创作模块是系统的核心功能模块，涵盖了歌曲播放、歌曲上传、歌词显示、播放记录等核心功能。该模块采用 MVC + Repository 分层架构设计，同时引入了播放历史记录和用户行为追踪机制，类与类之间的关系采用面向接口编程和依赖注入的设计模式。

继承关系：SongServiceImpl 实现了 SongService 接口，PlayerServiceImpl 实现了 PlayerService 接口。接口定义了歌曲和播放业务方法规范，实现类负责具体的歌曲 CRUD 操作、播放控制、播放历史记录等逻辑。这种设计遵循了面向接口编程的原则，使得业务逻辑的实现可以被替换，便于后期扩展和单元测试。

依赖关系：SongController 依赖 SongService 接口，PlayerController 依赖 PlayerService 接口，而不直接依赖实现类。控制层通过接口调用服务层，实现了控制层与服务层的解耦。当服务实现类发生变更时，控制器层无需修改，降低了代码耦合度。

关联关系：SongServiceImpl 通过依赖注入的方式关联了多个组件，包括 SongRepository（歌曲数据持久化）、ArtistRepository（艺术家数据持久化）、AlbumRepository（专辑数据持久化）、CategoryRepository（分类数据持久化）、SongArtistRepository（歌曲艺术家关联持久化）、PlayHistoryRepository（播放历史持久化）、UserBehaviorRepository（用户行为持久化）、UserRepository（用户数据查询）以及 FileStorageService（文件存储服务）。这些组件分别负责歌曲数据的增删改查、艺术家数据管理、专辑数据管理、分类数据管理、歌曲与艺术家关联关系管理、播放历史记录、用户行为记录、用户数据查询以及歌曲文件和封面图片的存储上传功能。PlayerServiceImpl 则关联了 PlayHistoryRepository、UserBehaviorRepository、SongRepository，用于记录播放历史和分析用户行为。

实体继承：Song、Artist、Album、Category、PlayHistory、UserBehavior 等实体类都继承自 JPA 的 BaseEntity（隐式），包含公共字段 id、createdAt、updatedAt。Song 实体类映射数据库中的 songs 表，存储歌曲的完整信息，包括歌曲标题、时长、文件URL、歌词、封面、专辑ID、分类ID、播放次数、下载次数、分享次数、状态、是否原创、上传用户ID等字段。PlayHistory 实体类映射 play_histories 表，记录用户的播放行为，包括用户ID、歌曲ID、播放时间、播放时长和状态。UserBehavior 实体类映射 user_behavior 表，记录用户对歌曲的详细行为，包括行为类型（播放、完整收听、收藏、分享、下载、评论）和权重分数。

数据传输：控制层接收客户端请求参数，包括歌曲ID、用户ID、播放进度等。服务层生成 Song、PlayHistory 等实体用于数据持久化，并返回 SongDTO、PlayHistoryDTO 等用于给前端展示。SongDTO 是专门用于展示歌曲信息的数据传输对象，包含歌曲ID、标题、艺术家（多个艺术家以逗号分隔）、专辑名称、分类名称、时长、封面URL、歌词、播放次数、分享次数、下载次数等信息。

数据流向：歌曲播放流程从客户端发起，GET 请求到 /api/song/play/{songId} 或 /api/player/play 接口，PlayerController 接收请求后，调用 PlayerServiceImpl 的 playSong 方法处理业务逻辑。服务层首先通过 SongRepository 查询歌曲信息，验证歌曲状态是否正常。然后通过 PlayHistoryRepository 记录用户的播放历史，通过 UserBehaviorRepository 记录用户的行为数据。最后返回歌曲的播放URL给客户端，客户端根据URL进行音频播放。

各层职责：控制层负责接收并解析客户端 HTTP 请求、参数校验、调用 Service 层方法、封装响应结果以及处理异常。服务层接口定义业务接口规范和声明业务方法签名，不涉及具体实现。服务层实现类负责实现核心业务逻辑，包括歌曲的搜索、详情查询、播放、上传、删除等操作，以及播放历史记录、用户行为追踪等功能。持久化层负责 Song、Artist、Album、Category、PlayHistory、UserBehavior 等实体的增删改查操作。实体层负责映射数据库表、定义实体属性。

### 7.2 时序图 - 播放歌曲

**流程说明：**

该时序图展示了用户播放歌曲的完整流程，整个流程分为验证歌曲、记录播放历史、记录用户行为和返回播放信息四个主要阶段。

验证歌曲阶段：客户端发起 GET 请求到 /api/player/play/{songId} 接口，请求播放指定歌曲。PlayerController 接收请求后，提取歌曲ID和用户ID参数，调用 PlayerServiceImpl 的 playSong 方法处理业务逻辑。服务层首先通过 SongRepository 根据歌曲ID查询歌曲信息，验证歌曲是否存在。如果歌曲不存在，返回错误信息给客户端。如果歌曲存在，继续验证歌曲状态是否正常（status=1表示正常）。如果歌曲已禁用，返回错误信息。如果歌曲正常，获取歌曲的播放URL和相关信息。

记录播放历史阶段：歌曲验证通过后，服务层通过 PlayHistoryRepository 记录用户的播放历史。首先创建一个 PlayHistory 实体对象，设置用户ID、歌曲ID、播放时间（当前时间）、播放时长（从歌曲元数据获取）和状态。然后将播放历史记录保存到数据库。播放历史记录用于后续的用户听歌报告生成和个人播放统计。

记录用户行为阶段：播放历史记录保存成功后，服务层通过 UserBehaviorRepository 记录用户的行为数据。创建一个 UserBehavior 实体对象，设置用户ID、目标类型（歌曲为1）、目标ID（歌曲ID）、行为类型（播放为1）、权重分数（播放权重为1）和创建时间。然后将行为记录保存到数据库。用户行为数据用于用户兴趣分析和推荐系统。

返回播放信息阶段：服务层将歌曲播放相关信息封装为 Map 对象返回给客户端。返回信息包括歌曲ID、歌曲标题、艺术家名称、专辑名称、封面URL、播放URL、歌词（如果需要）、播放时长等。Controller 最终返回 200 OK 响应给客户端，响应体中包含 code 为 200 和 data 为播放信息的 JSON 对象。客户端收到响应后，根据播放URL开始音频播放。

### 7.3 时序图 - 上传歌曲

**流程说明：**

该时序图展示了用户上传歌曲的完整流程，整个流程分为接收文件、存储文件、创建歌曲记录、关联艺术家和返回结果五个主要阶段。

接收文件阶段：客户端发起 POST 请求到 /api/song/upload 接口，请求上传歌曲文件。请求采用 multipart/form-data 格式，包含歌曲文件（file 参数）和其他元数据（title、duration、artistName、albumName、categoryId、lyrics 等）。SongController 接收请求后，通过 @RequestParam 注解获取歌曲文件和其他参数，调用 SongServiceImpl 的 uploadSong 方法处理业务逻辑。服务层首先验证上传的文件是否为空，如果为空则返回错误信息。然后验证文件类型是否为支持的音频格式（mp3、wav、flac、m4a等）。

存储文件阶段：文件验证通过后，服务层通过 FileStorageService 存储上传的歌曲文件。FileStorageService 会为文件生成唯一的文件名（通常使用 UUID + 原始扩展名），将文件保存到配置的存储目录，并返回文件的访问URL。服务层获取到歌曲文件的存储URL后，如果客户端也上传了封面图片，同样通过 FileStorageService 保存封面图片，获取封面URL。

创建歌曲记录阶段：文件存储成功后，服务层需要创建歌曲记录到数据库。首先创建 Song 实体对象，设置歌曲标题（从请求参数获取）、时长（从请求参数或解析文件获取）、歌曲文件URL、封面URL（如果有）、歌词（如果有）、分类ID、专辑ID（如果指定了已有专辑）、上传用户ID、歌曲状态（默认1为正常）、是否原创（用户上传默认为0）等字段。然后通过 SongRepository 将歌曲保存到数据库，获取自动生成的歌曲ID。

关联艺术家阶段：歌曲记录创建成功后，服务层需要处理歌曲与艺术家的关联关系。如果请求中指定了艺术家名称，系统首先通过 ArtistRepository 查询该艺术家是否已存在。如果艺术家不存在，则创建一个新的 Artist 记录。如果艺术家已存在，获取艺术家的ID。然后通过 SongArtistRepository 创建歌曲与艺术家的关联关系（一首歌曲可以有多个艺术家，如合唱歌曲）。如果是多艺术家的情况，需要分别创建多条关联记录。

返回结果阶段：服务层将创建的歌曲信息转换为 SongDTO 格式返回给客户端。返回信息包括歌曲ID、歌曲标题、艺术家名称、专辑名称、分类名称、时长、封面URL、歌词、播放次数（初始为0）等。Controller 最终返回 201 Created 响应给客户端，响应体中包含 code 为 201 和 data 为歌曲信息的 JSON 对象。

### 7.4 时序图 - 获取播放历史

**流程说明：**

该时序图展示了用户获取自己播放历史的完整流程，整个流程分为查询播放历史、获取歌曲详情和返回结果三个主要阶段。

查询播放历史阶段：客户端发起 GET 请求到 /api/player/history/{userId} 接口，请求获取指定用户的播放历史记录。PlayerController 接收请求后，提取用户ID参数，调用 PlayerServiceImpl 的 getPlayHistory 方法处理业务逻辑。服务层首先通过 PlayHistoryRepository 查询该用户的播放历史记录。查询使用 findByUserIdOrderByPlayTimeDesc 方法，按照播放时间降序排列，即最新的播放记录排在最前面。如果需要限制返回数量，可以使用 findByUserIdAndPlayTimeAfter 方法查询指定时间段内的播放记录。

获取歌曲详情阶段：查询到播放历史记录列表后，服务层需要为每条历史记录补充歌曲的详细信息。因为 PlayHistory 实体只存储了歌曲ID，不包含歌曲的标题、艺术家、封面等信息。服务层遍历播放历史列表，对每条记录通过 SongRepository 查询对应的歌曲信息。然后将歌曲信息（标题、艺术家名称、封面URL等）填充到 PlayHistoryDTO 中。如果歌曲已被删除或不存在，需要处理这种情况，可以返回默认信息或者跳过该条记录。

返回结果阶段：服务层将包含歌曲详情的播放历史列表返回给客户端。返回数据为 List<PlayHistoryDTO> 格式，每条记录包含历史记录ID、歌曲ID、歌曲标题、艺术家名称、封面URL、播放时间和播放时长等信息。Controller 最终返回 200 OK 响应给客户端，响应体中包含 code 为 200 和 data 为播放历史列表的 JSON 对象。

### 5.2 时序图 - AI 音乐生成（异步任务）

**流程说明：**

该时序图展示了AI音乐生成的完整异步任务流程，整个流程分为四个主要阶段，包括任务提交、后台异步执行、轮询等待和状态查询。

阶段一为提交生成任务。客户端发起 POST 请求到 /api/ai/generate 接口，请求生成AI音乐，请求参数包括 prompt（生成提示词，如"欢快的歌曲"）、genre（音乐风格，如"流行"）、mood（音乐情绪，如"欢快"）、duration（生成时长，如60秒）。AiMusicController 接收请求后，调用 AiMusicService 的 generateMusicAsync 方法处理业务逻辑。服务层首先创建一个 AiMusicGeneration 实体对象，设置状态为0（表示生成中），然后通过 AiMusicGenerationRepository 将生成记录保存到数据库。保存成功后，服务层返回一个 Future 对象给 Controller，Controller 立即返回 202 Accepted 响应给客户端，响应中包含任务信息和状态为 processing。这种异步处理模式允许客户端在等待AI音乐生成的过程中执行其他操作，客户端可以通过返回的任务ID轮询查询生成状态。

阶段二为后台异步执行。线程池从任务队列中取出异步任务开始执行，调用 AiMusicService 的 callMusicGenService 方法。服务层构造HTTP请求，调用外部AI音乐生成服务（MusicGen Service）的 POST /api/generate 接口，请求参数包括提示词和生成时长。外部服务接收到请求后，创建一个异步生成任务，立即返回一个任务ID（如 task_id=xxx）给服务层。此时AI音乐生成任务已在后台开始处理，服务层获得任务ID后返回等待进一步处理。

阶段三为轮询等待生成完成。服务层获得任务ID后，开始轮询外部AI音乐服务的任务状态。服务层循环调用 GET /api/status/{task_id} 接口查询任务进度，每次间隔5秒，最多轮询120次。如果任务状态仍为 processing（处理中），继续等待下一轮轮询。当某次查询返回状态为 completed（已完成）并附带生成的音频URL时，说明AI音乐已生成完毕。服务层调用 downloadAndSaveAudio 方法下载生成的音频文件，并保存到本地存储路径。然后更新数据库中的生成记录，将状态修改为1（表示成功），并保存生成的音乐URL。如果在轮询过程中发生错误或超时，服务层会将状态修改为2（表示失败），并保存错误信息。

阶段四为查询任务状态。客户端可以通过 GET /api/ai/generate/{id} 接口主动查询任务的最新状态。AiMusicController 接收请求后，调用 AiMusicService 的 getGenerationStatus 方法。服务层通过 AiMusicGenerationRepository 根据任务ID查询生成记录。根据记录的状态返回不同的响应给客户端：如果状态为1（成功），返回200 OK和包含音乐URL的完整记录；如果状态为0（进行中），返回200 OK和当前进度信息，客户端可以继续轮询；如果状态为2（失败），返回200 OK和错误信息，客户端可以查看失败原因。

### 7.2 时序图 - 获取全局热门榜单

**流程说明：**

该时序图展示了客户端获取全局热门榜单的完整流程，整个流程分为缓存查询、数据处理和响应返回三个主要阶段。

缓存查询阶段：客户端发起 GET 请求到 /api/ranking/global 接口，请求获取全局热门榜单数据。RankingController 接收请求后，调用 RankingServiceImpl 的 getGlobalRanking 方法处理业务逻辑。服务层首先从缓存中查询全局热门榜单数据，通过 HotRankingRepository 的 findGlobalByRankingType 方法查询排名类型为"global"的榜单数据。如果查询结果不为空，说明缓存中存在有效的榜单数据，系统将直接使用这些数据进行后续处理。

数据处理阶段：系统根据缓存查询的结果分为两种处理路径。如果缓存中有数据，系统直接进行 DTO 构建，不需要重新生成榜单。如果缓存中没有数据（缓存未命中），系统需要调用 generateGlobalRanking 方法生成新的榜单。生成过程首先通过 HotRankingRepository 的 deleteByRankingType 方法删除旧的榜单数据，然后从 SongHotScoreRepository 查询热度分数排名前100的歌曲，接着循环构建 HotRanking 实体并保存到数据库，最后再次查询生成的新榜单数据。

响应返回阶段：无论是从缓存获取还是重新生成，服务层都需要将榜单数据转换为 DTO 格式返回给客户端。系统调用 buildHotSongsFromRankings 方法构建歌曲列表，对于榜单中的每个排名，通过 SongRepository 根据歌曲ID查询歌曲的详细信息，包括歌曲标题、封面、艺术家等。然后调用 buildHotSongDTO 方法将每首歌曲的信息封装为 HotSongDTO 对象。完成所有歌曲的 DTO 转换后，服务层将包含完整榜单信息的 HotRankingDTO 返回给 Controller，Controller 最终返回 200 OK 响应给客户端，响应体中包含 code 为 200 和 data 为榜单数据的 JSON 对象。

    rect rgb(230, 245, 255)

        RankingServiceImpl->>HotRankingRepository: findGlobalByRankingType("global")
        HotRankingRepository-->>RankingServiceImpl: List~HotRanking~
    end

    alt 有缓存数据
        Note over RankingServiceImpl,RankingServiceImpl: 缓存命中，直接返回
    else 无缓存数据

        RankingServiceImpl->>RankingServiceImpl: generateGlobalRanking()
        
        rect rgb(255, 245, 230)
            Note over RankingServiceImpl,HotRankingRepository: 删除旧榜单
            RankingServiceImpl->>HotRankingRepository: deleteByRankingType("global")
        end
        
        rect rgb(245, 255, 230)
            Note over RankingServiceImpl,SongHotScoreRepository: 按热度分数排序
            RankingServiceImpl->>RankingServiceImpl: 查询热度分数Top100
            RankingServiceImpl->>RankingServiceImpl: 循环保存排名
            RankingServiceImpl->>HotRankingRepository: save(ranking)

        Note over RankingServiceImpl,PlayHistoryRepository: 计算播放分数
        loop 每首歌曲
            RankingServiceImpl->>PlayHistoryRepository: countBySongId(songId)
            PlayHistoryRepository-->>RankingServiceImpl: playCount
            RankingServiceImpl->>RankingServiceImpl: playScore = playCount * 1.0
        end
    end

    rect rgb(245, 255, 230)
        Note over RankingServiceImpl,UserBehaviorRepository: 计算收藏分数
        loop 每首歌曲
            RankingServiceImpl->>UserBehaviorRepository: countBySongIdAndBehaviorType(songId, 3)
            UserBehaviorRepository-->>RankingServiceImpl: favoriteCount
            RankingServiceImpl->>RankingServiceImpl: favoriteScore = favoriteCount * 5.0
        end
    end

    rect rgb(245, 230, 255)
        Note over RankingServiceImpl,UserBehaviorRepository: 计算分享分数
        loop 每首歌曲
            RankingServiceImpl->>UserBehaviorRepository: countBySongIdAndBehaviorType(songId, 4)
            UserBehaviorRepository-->>RankingServiceImpl: shareCount
            RankingServiceImpl->>RankingServiceImpl: shareScore = shareCount * 3.0
        end
    end

    rect rgb(230, 255, 245)
        Note over RankingServiceImpl,CommentRepository: 计算评论分数
        loop 每首歌曲
            RankingServiceImpl->>CommentRepository: countBySongId(songId)
            CommentRepository-->>RankingServiceImpl: commentCount
            RankingServiceImpl->>RankingServiceImpl: commentScore = commentCount * 2.0
        end
    end

    rect rgb(255, 245, 255)
        Note over RankingServiceImpl,RankingServiceImpl: 计算综合热度分数
        loop 每首歌曲
            RankingServiceImpl->>RankingServiceImpl: hotScore = playScore + favoriteScore + shareScore + commentScore
            Note right of RankingServiceImpl: 权重：播放*1 + 收藏*5 + 分享*3 + 评论*2
        end
        
        RankingServiceImpl->>SongHotScoreRepository: saveAll(scores)
        SongHotScoreRepository-->>RankingServiceImpl: saved
    end

    Note right of RankingServiceImpl: 热度分数计算完成
